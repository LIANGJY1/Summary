# Harness Engineering 核心解析：决定 AI Agent 成败的关键系统

**Q: 为什么同样的模型，别人做的 Agent 成功率高且稳定，而自己的却差强人意？**

**A**: 很多时候问题并不在于模型本身或提示词，而在于模型外部的运行系统。这套运行系统现在有一个统一的名称——**Harness Engineering**（或 Harness）。**优化任务拆解、状态管理、关键步骤校验以及失败恢复等机制**，往往能将任务成功率从不足 70% 大幅提升至 95% 以上。

***

## 一、 AI 工程的三次演进

**Q: 过去两年，AI 工程经历了哪三次重心迁移？**

**A:** AI 工程经历了三个阶段的演进，分别对应解决不同层面的问题：

1. **Prompt Engineering（提示词工程）**：解决模型“**有没有听懂**”的问题。
2. **Context Engineering（上下文工程）**：解决模型“**有没有拿到足够且正确的信息**”的问题。
3. **Harness Engineering（Harness 工程）**：解决模型“**在真实执行中能不能持续做对**”的问题。

**Q: 什么是 Prompt Engineering（提示词工程）？它有什么局限性？**
A: 提示词工程的核心是塑造局部的概率空间。通过设定角色、风格、示例（few-shot）和约束，引导模型输出期望的结果。

- **擅长**：长期任务约束输出，激发模型的已有能力。
- **局限性**：无法凭空弥补缺失的知识、难以处理大量动态信息，也不擅长管理长链路任务中的状态。它解决的是表达意图的问题，而非信息供给的问题。

**Q: 什么是 Context Engineering（上下文工程）？它与 RAG 有什么关系？**
A: 上下文工程的核心是：系统必须在合适的时机将正确的信息输入给模型。上下文（Context）不仅包含提示词，还包括用户输入、历史对话、检索结果、工具返回、任务状态及系统规则等。

- RAG（检索增强生成）是其典型实践，用于在运行时补充知识。
- 更高级的实践（如 Agent Skills 的渐进式披露）强调按需、分层、在正确时机提供信息，以避免信息过载导致模型注意力涣散。

**Q: 什么是 Harness Engineering？它与前两者的区别是什么？**
A: Harness 的本意是“缰绳、约束装置”。在 AI 系统中，Harness 意味着系统不仅要喂给模型信息，还要能驾驭整个执行过程。

- 如果 Prompt 是指令工程化，Context 是输入环境工程化，那么 Harness 就是**整个运行系统的工程化**。
- LangChain 提出了一个公式：`Agent = Model + Harness`。除了模型本身，所有决定系统能否稳定交付的组件，都属于 Harness。它关注的是如何让模型不跑偏、跑得稳，并在出错时能拉回正轨。

## 二、 成熟 Harness 的六个层级

**Q: 一个成熟的 Harness Engineering 包含哪六个层级？**

**A: 第一层：Context 视角（信息边界控制）**
确保模型在正确的信息边界内思考，包括：

1. 明确角色的目标、任务和成功标准。
2. 裁剪冗余信息，只保留高相关性上下文。
3. 结构化组织信息（规则、状态、证据分层存放），避免模型遗漏重点或自我污染。

**第二层：工具系统**
大模型需要工具才能连接真实世界。Harness 需要解决：

1. 提供合适的工具集合（避免过少导致能力不足，过多导致乱用）。
2. 规定调用时机（何时该查证，何时该直接应答）。
3. 对工具返回的结果进行提炼筛选，保持与任务的相关性。

**第三层：执行编排**
解决“下一步该做什么”的问题，避免模型想到哪做到哪。完整的任务轨道应包括：理解目标 -> 评估信息并补充 -> 分析结果 -> 生成并检查输出 -> 修正或重试。

**第四层：记忆和状态**
管理 Agent 的状态，防止“失忆”或逻辑混乱。需要清晰区分并管理三类数据：

1. 当前任务的状态。
2. 会话中的中间结果。
3. 长期记忆和用户偏好。

**第五层：评估和观测**
赋予系统独立评估结果好坏的能力，避免 Agent 陷入“自我感觉良好”。包括：输出验证、自动化测试、日志指标监控和错误归因。

**第六层：约束、校验、失败与恢复（上线关键环节）**
真实环境中失败是常态，必须建立兜底机制：

1. **约束**：明确能力边界（能做与不能做）。
2. **校验**：输出前后的自动检查。
3. **恢复**：失败后的重试策略与状态回滚机制。

## 三、 一线公司的 Harness 实践案例

**Q: Anthropic 在构建自主编码系统时，是如何解决上下文溢出和自我评估失真问题的？**
A:

1. **解决“上下文焦虑” (Context Reset)**：
   当上下文过长导致模型丢失细节时，仅做上下文压缩（Compression）效果有限。Anthropic 采用了 Context Reset 策略——直接更换一个干净的全新 Agent 并交接工作，类似于工程中的“重启进程恢复状态”。
2. **解决自评失真 (分离干活与验收)**：
   让模型自己打分往往偏乐观。Anthropic 将干活与验收的角色分离：
   - **Planner**：扩展需求规格。
   - **Generator**：逐步实现代码。
   - **Evaluator**：像 QA 一样进行真实环境测试（不仅看代码，还操作页面检查交互结果）。
     这贯彻了“生产与验收必须分离”的工程原则，形成有效的生成、检查、修复循环。

**Q: OpenAI 在构建全自动 Agent 应用时，工程师的角色发生了什么变化？他们有哪些典型的 Harness 实践？**
A: 工程师的角色从“写代码”转变为“设计环境”。具体实践包括：

1. **重定义修复方案**：当 Agent 失败时，不应要求它“更努力”，而是分析环境缺乏什么能力结构，并建立反馈链路让 Agent 看到真实结果。
2. **渐进式披露 (Skills 思维)**：摒弃将所有规范塞入单一庞大文档的做法，改为提供“目录页”。Agent 先看索引，需要时再去钻研具体的子文档（如架构、设计、安全规则等），避免上下文窗口被冗余信息挤占。
3. **让 Agent “看见”并自我验证**：赋予 Agent 浏览器截图、模拟操作、查日志和监控的能力，并在隔离环境中运行。Agent 需独立完成跑代码、查 Bug、修 Bug 和再验证的全流程。
4. **将专家经验转化为自动治理系统**：将资深工程师的 Code Review 经验（如模块分层、拦截条件）写成系统规则。当系统报错时，不仅提示错误，还附带修复建议一并反馈给 Agent，实现可持续的自动治理。
